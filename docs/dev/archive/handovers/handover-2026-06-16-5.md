## ⛔ MODEL LAW locked this session (in modifier-spec §5 + renames §Promotion/§UnitCombat)

- **THE §5 UNIT-STAT VOCABULARY — DEFINED (shared by Promotion + UnitCombat + SpecialUnit + Unit):** all `unit`-scope
  self-accumulator deposits. **`strength`** = THE combat family ("the strength of something, or weakness on/into/against
  something", owner) — general %/flat/SM(sizeModifier·quality·group·size·maxHP·perSize·perVolume)/situational(city·hills)/
  vs-keyed(terrain·feature·unitCombat·domain·flanking) + the S&D(unnerve·enclose·lunge·dynamicDefense, LIVE option
  `SURROUND_DESTROY`) & TB sub-stats. Plus `withdrawal`/`firstStrike`/`bombard`/`collateral`/`air`/`heal`/`movement`/
  `experience`/`workRate`/`cargo`/`upkeep`/`vision`/`capture`/`poison`/`espionage`/`trap` + singletons (`revoltProtection`/
  `pillage`/`survivor`). per-`PROPERTY_*` via `property_source_v3`.
- **CAPABILITIES = a SEPARATE group of BOOLEANS** (owner): grant=`true`/revoke=`false` (the Add/Subtract & Remove pairs);
  the `>0` count-ints + per-type ability maps live here too. Pure abilities, NOT magnitude modifiers (§3).
- **`capture` is a GRADIENT, its OWN family** (probability/resistance; the engine nets prob−resist → a 0-99% chance,
  summed across unit/promotion/commander/national/local). Captives + subdued animals; ⚑ owner intent: capture vs subdue
  *should* split later (a hunter trapping animals shouldn't take military captives) — not changed in #428.
- **VISION/LOS = a non-cascade RESOLVER sub-object** (§7): the 2D Invisible/Visible{Terrain,Feature,Improvement}[Range]
  tables + Visibility/InvisibilityIntensity[Range/SameTile] + negates → `vision` block, read by the visibility resolver.
- **properties → SCOPED MODIFIER deposits** (`PROPERTY_X.{plot,city}.flat`): the crime/disease/education unit→city/plot
  emission, "scoped like other property yields, like a modifier" (owner). **`engine.property_source_v3` extended to
  accept `RELATION_SAME_PLOT` as the containment default** (alongside `ASSOCIATED`); scope from `GameObjectType`. A
  promotion emitting to both = TWO modifiers (plot+city). (UnitCombat carries no property data — empty/dead, §8-i.)
- **`promotionLine: {LINE: rank}` OBJECT** (owner): line membership keyed by line → rank (`iLinePriority` is a RANK, not
  a priority). The OBJECT shape is ACCUMULATOR-FRIENDLY — a unit's promotions MERGE their maps into one `{LINE: rank,…}`
  → a clean **bottom→up rank overwrite + line-hierarchy check**. ~261 of 1229 have NO line.
- **THE STATIC-DEF / CONCATENATION PRINCIPLE (owner):** the promotion/unitcombat JSON is the STATIC DEFINITION, NOT the
  on-a-unit instance — but every shape is authored for **O(1) runtime CONCATENATION** onto the unit (sum `unit`-scope
  deposits, union capabilities, merge `promotionLine`), never apply-time post-processing.
- **`grants` (NOW) vs `outcomes` (FUTURE) — a deliberate distinction (owner):** `grants` fires immediately on its event
  IF the enabler/tech is held (§6.1 enable→grant-check). `outcomes` are DEFERRED — parked on the unit (ACCUMULATED),
  fired only when the MISSION is pushed. Same one-time-yield SHAPE (the commerce burst is `AdaptUnitYield`-wrapped =
  GameSpeed `missionYieldMultiplier`, the SAME scale a merchant trade-mission uses), different LIFECYCLE → `outcomes` is
  its own section, NOT folded into `grants`.
- **`*Base` ranks = create-unit data (§0.6), kept as-is** → `identity.base`. **Size Matters mostly governs in UnitCombat**
  (owner) → see the deferred SM-module pass below.
